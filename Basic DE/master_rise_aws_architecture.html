Kiến trúc dữ liệu · Master Rise Group (AWS)

Data Platform trên AWS: kiến trúc lakehouse điển hình

Cũng là mô hình ELT giống OneMount, nhưng dựng bằng bộ dịch vụ managed của AWS thay vì tự vận hành trên Kubernetes/GCS. Cấu trúc 3 lớp Bronze → Silver → Gold và vai trò của Iceberg giữ nguyên như bản GCP.

Lưu ý Đây là kiến trúc tham khảo — dựng theo một pattern lakehouse phổ biến trên AWS (S3 + Iceberg + Glue + Athena), chưa xác nhận với hệ thống thật của Master Rise Group. Cần đối chiếu lại tên service cụ thể trước khi dùng làm tài liệu chính thức.
Nguồn

Source Systems

Database OLTP của các app nghiệp vụ, event tracking web/mobile, log hệ thống — giống vai trò như ở kiến trúc GCP.

Nói đơn giản: nơi dữ liệu được tạo ra lần đầu.

Message bus

Amazon MSK (hoặc Kinesis Data Streams)

MSK là Kafka được AWS vận hành hộ (managed) — cùng khái niệm topic/partition/consumer group như Kafka tự triển khai. Kinesis Data Streams là lựa chọn thay thế "AWS-native", cùng vai trò nhưng API và mô hình scaling khác (shard thay vì partition).

Nói đơn giản: vẫn là "hàng đợi trung gian" giữa nguồn và job xử lý — chỉ khác ai là người vận hành cụm Kafka.

Ingest & land · managed

Kinesis Data Firehose (hoặc EMR/Glue Streaming job)

Firehose là dịch vụ managed, không cần tự viết job Spark: nó tự buffer dữ liệu theo thời gian/kích thước, và có thể tự convert JSON sang Parquet ngay khi ghi xuống S3 nếu khai báo schema trước trong Glue Data Catalog. Nếu cần logic biến đổi phức tạp hơn khi đang ingest, dùng EMR hoặc Glue Streaming job (tương đương vai trò Spark Structured Streaming ở bản GCP).

Nói đơn giản: đây là "máy bơm" liên tục đổ dữ liệu từ hàng đợi vào kho lưu trữ — Firehose là bản không cần tự vận hành server, EMR/Glue là bản tự kiểm soát logic nhiều hơn.

S3 Data Lake · Bronze

Bronze — Raw JSON / Raw Parquet

Vùng lưu dữ liệu gần nguyên bản, append-only, partition theo ngày/giờ ingest. Nếu Firehose không convert Parquet ngay từ bước trước, vẫn cần một job Glue/EMR batch chạy định kỳ để gộp file nhỏ và chuẩn hoá sang Parquet — cùng lý do như bản GCP (chi phí liệt kê file trên S3, tốc độ đọc row-based vs columnar).

Nói đơn giản: "bản sao y nguyên" dữ liệu vừa nhận, giữ lại để có thể chạy lại nếu bước sau lỗi.

Batch / micro-batch

AWS Glue ETL Jobs (hoặc EMR Spark)

Glue ETL job (Spark chạy serverless, AWS tự quản lý cluster) hoặc EMR (cluster Spark tự cấu hình): khử trùng lặp, join dữ liệu tham chiếu, kiểm tra chất lượng, áp business rule, ghi kết quả xuống Silver dưới dạng bảng Iceberg.

Nói đơn giản: bước "làm sạch và ráp nối" — giống hệt vai trò Spark ETL Jobs ở bản GCP, chỉ khác ai host cụm Spark.

S3 · Silver / Gold

Iceberg Tables trên S3

Silver: dữ liệu chi tiết đã sạch. Gold: bảng tổng hợp sẵn cho báo cáo. Catalog của Iceberg ở đây là AWS Glue Data Catalog — đóng đúng vai trò catalog đã nhắc ở bản GCP (biết bảng nào đang trỏ tới metadata.json nào), và tương thích Hive Metastore API nên hầu hết engine trong hệ AWS đều đọc được.

Nói đơn giản: Silver là dữ liệu chi tiết đã sạch, Gold là số liệu tổng hợp sẵn — vẫn là file Parquet, chỉ khác nơi lưu catalog.

Query engine

Amazon Athena (hoặc Redshift Spectrum)

Athena là engine SQL serverless, trả phí theo dữ liệu quét — không cần dựng cluster, đọc thẳng bảng Iceberg qua Glue Data Catalog. Redshift Spectrum là lựa chọn thay thế khi cần join dữ liệu lake với data warehouse Redshift đã có sẵn. Khác với Dremio, Athena không có tầng "reflection" cache riêng — muốn tăng tốc truy vấn lặp lại thường dựng thêm bảng Gold tổng hợp sẵn hoặc dùng Redshift làm lớp serving phía sau.

Nói đơn giản: đây là nơi Power BI/Data Scientist "hỏi" dữ liệu bằng SQL — trả tiền theo lượng dữ liệu quét, không phải theo giờ chạy máy chủ.

BI

Amazon QuickSight

Dashboard, báo cáo định kỳ, kết nối trực tiếp tới Athena hoặc Redshift.

Phân tích / ML

Data Scientist (SageMaker)

Truy vấn ad-hoc qua Athena, hoặc đọc thẳng Iceberg bằng Spark (EMR) khi cần xử lý khối lượng lớn cho feature engineering, huấn luyện mô hình trong SageMaker.

Khác biệt đáng chú ý so với bản GCP

Không phải đổi tên service — có vài chỗ đổi cả cách vận hành

Phần lớn các bước tương ứng 1-1 với bản OneMount (Kafka↔MSK, GCS↔S3, Spark ETL↔Glue/EMR, Iceberg vẫn là Iceberg). Nhưng có ba điểm khác đáng lưu ý khi làm việc với bản AWS:

Điểm khácÝ nghĩa vận hành
Firehose có thể tự convert Parquet Nếu dùng Firehose thay vì tự viết Spark Streaming, bước "node convert" ở Bronze có thể nằm ngay trong Firehose (khai schema trước trong Glue Catalog), không cần job batch riêng — nhưng đổi lại ít linh hoạt hơn khi cần logic transform phức tạp lúc ingest.
Glue Data Catalog là catalog bắt buộc trên AWS Không có lựa chọn "Nessie" như thường thấy ở triển khai tự quản — Glue Data Catalog gần như mặc định vì mọi service AWS (Athena, EMR, Redshift Spectrum, Glue ETL) đều đọc được nó.
Athena tính phí theo dữ liệu quét, không có reflection Không tối ưu partition/compaction tốt ở lớp Gold thì chi phí Athena tăng trực tiếp theo lượng dữ liệu quét — khác với Dremio, nơi reflection có thể che bớt phần này.
Kiến trúc tham khảo cho Master Rise Group, dựng theo pattern lakehouse phổ biến trên AWS (Kinesis/MSK → S3 Bronze → Glue/EMR → S3 Iceberg Silver/Gold → Athena → QuickSight/SageMaker). Cần xác nhận lại với hệ thống thật trước khi dùng làm tài liệu chính thức.
