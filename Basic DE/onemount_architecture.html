Chỗ chưa rõ #1

Vì sao lại có node "convert raw → Parquet" ở Bronze?

Spark Structured Streaming ưu tiên ghi nhanh, không chặn: mỗi micro-batch (vài giây một lần) tạo ra một file nhỏ, thường ở dạng Avro/JSON — định dạng theo dòng (row-based), ghi rẻ, hỗ trợ schema evolution tốt, phù hợp cho việc "hứng" dữ liệu liên tục. Nhưng ghi liên tục như vậy tạo ra rất nhiều file nhỏ, mà file nhỏ + định dạng theo dòng thì lại chậm khi các job ETL/BI phía sau cần đọc và quét dữ liệu.

Vì vậy có một job Spark chạy định kỳ (ví dụ mỗi giờ), làm hai việc cùng lúc: gộp nhiều file nhỏ thành file lớn (compaction) và chuyển từ Avro sang Parquet — định dạng theo cột (columnar), nén tốt hơn, và cho phép đọc chỉ những cột cần dùng thay vì cả dòng. Đây chính là "node convert" mà bên DE nhắc tới: nó là cầu nối giữa vùng hạ cánh thô (tối ưu cho ghi) và Bronze chuẩn hoá (tối ưu cho đọc), để mọi job phía sau chỉ cần biết một định dạng duy nhất.

Streaming ghi ra Nhiều file nhỏ .avro Row-based, ghi mỗi vài giây, tối ưu cho tốc độ & schema evolution
→
Job convert + compaction Spark batch job Gộp file nhỏ lại, đọc Avro, ghi lại thành Parquet
→
Bronze chuẩn hoá Ít file lớn .parquet Columnar, nén tốt, sẵn sàng cho Spark ETL đọc nhanh
Chỗ chưa rõ #2

Iceberg "quản" các file Parquet như thế nào?

Điểm mấu chốt: Iceberg không thay thế Parquet, cũng không phải nơi lưu dữ liệu. File dữ liệu thật sự vẫn là các file .parquet bình thường nằm trong GCS — y hệt như trước. Iceberg chỉ là một lớp metadata ngồi phía trên, ghi nhớ: "table này gồm đúng những file Parquet nào, ở version nào". Nó biến một đống thư mục Parquet rời rạc thành một bảng dữ liệu có version, có transaction.

catalog (Dremio / Hive Metastore / Nessie — biết bảng đang trỏ tới đâu)
└─ metadata.json — schema hiện tại, danh sách snapshot, snapshot nào là "current"
└─ snapshot (mỗi lần ghi/commit mới = 1 snapshot mới)
└─ manifest list — danh sách các manifest file thuộc snapshot này
└─ manifest file — danh sách data file + thống kê (min/max, số dòng…)
└─ data-001.parquet, data-002.parquet, … ← file thật, nằm trên GCS

Nhờ lớp metadata này, khi Dremio hay Spark truy vấn, engine không cần liệt kê toàn bộ thư mục trên GCS (vốn chậm và tốn chi phí) — nó đọc thẳng manifest để biết chính xác cần mở file Parquet nào, bỏ qua file nào. Và vì mỗi lần ghi tạo ra một snapshot mới thay vì sửa trực tiếp file cũ, Iceberg có được: ghi/đọc không đụng nhau (ACID), đổi schema hoặc partition mà không phải viết lại toàn bộ dữ liệu, và có thể "time travel" — truy vấn lại đúng trạng thái bảng tại một thời điểm trước đó.
